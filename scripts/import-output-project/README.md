# import-output-project

Turn an **existing analyzer output folder** (generated by a past pipeline run) into a
project that is browsable in the web UI — **without re-running the pipeline**.

It creates a Project + Version + Commit + Documents (+ section bodies) in the JSON-backed DB
(`api/db/data/*.json`) and copies the artifacts into the
`workspaces/<pid>/<commit[:16]>/` layout the render route reads.

## Usage

```bash
# Point at a per-commit snapshot dir (auto-discovers output/, model/, and config.json):
python scripts/import-output-project/import_output_project.py \
    workspaces/pb701836d/08d2f565cd03e72e --name "Imported Demo"

# Or point at a bare output/ folder and supply the rest explicitly:
python scripts/import-output-project/import_output_project.py output \
    --name "My Project" --model model --config config/config.json
```

Then start the API against the JSON store and open the web app:

```bash
API_DB_BACKEND=json uvicorn api.main:app --reload
(cd web-app && npm run dev)
```

Sign in with a seeded user (e.g. `admin@aspice.dev` / `secret`) and open the new project.

## What it reads

`source` may be a **snapshot dir** (contains `output/` and usually `model/`, with the
project `config.json` one level up) or the **`output/` folder** itself.

| Thing | Auto-discovery | Override |
|-------|----------------|----------|
| output root | `source/output`, else `source` | `--output DIR` |
| model dir | `source/model`, else `<output>/../model` | `--model DIR` |
| config (layers) | `source/config.json`, `source/../config.json`, … | `--config FILE` |
| project name | model `metadata.json → projectName`, else folder name | `--name` |
| tag / branch / commit | `v1.0.0` / `main` / synthesized sha | `--tag` `--branch` `--commit` |
| compliance / repo-url / client | `ISO_26262` / `""` / `""` | `--compliance` `--repo-url` `--client` |
| owner | `admin@aspice.dev` (admin member + creator) | `--owner-email` |
| document status | `approved` (also marks version/commit approved, project complete) | `--status {approved,in_review,never,unchanged}` |

The document **layer/group labels** come from the analyzer `config.json` `layers` tree when
found; groups it doesn't cover fall back to a single synthesized layer (`--layer`, default
`LAYER1`). Only output subdirs that actually hold
`software_detailed_design_<group>.docx` become documents.

## Notes

- Writes to the JSON DB, so the API must run with `API_DB_BACKEND=json` to see the project.
- Supplying `--model` (or a snapshot with a `model/` dir) makes functions/units/data-dictionary
  render correctly for this version. The render route reads the imported project's own
  `workspaces/<pid>/<commit[:16]>/model` **only because imported projects have no `repo_url`**
  — real repo-backed projects are unaffected and keep reading the shared repo `model/`. So for
  full model fidelity, import from a snapshot that includes its `model/` (or pass `--model`).
- Records are built with the pipeline's own `_make_documents` / `_make_sections`, so an
  imported project is indistinguishable from one produced by a real run.
- Re-running creates a **new** project id each time (safe to run repeatedly).
